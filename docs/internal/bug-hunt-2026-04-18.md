## Release-critical follow-up — AI budget architecture

Current design caps AI calls per tick (`max_ai_calls_per_tick=5`) which is
a **throughput** cap, not a **budget** cap. Under a coordinated attack of
N simultaneous Calderas (observed during this exercise: ~20-30 incidents
per minute), the cap just defers incidents into a growing backlog — so
a real attacker mixed into the noise can evade detection long enough to
get in.

The economics invert the intended safeguard: saving a few dollars of AI
spend to miss a compromise that costs thousands to clean up is the wrong
trade. The code already implements the right primitive (allowlist +
cooldown per `action:detector:entity`) to prevent runaway hammering on
the same incident; what is missing is a **budget-aware tiering** on top.

Desired architecture (spec-worthy, not yet implemented):

1. **Config**: replace `max_ai_calls_per_tick` with `ai_budget_usd_per_day`
   — default generous (e.g. $50) and tracked in the existing telemetry
   counter alongside `ai_provider_errors_per_hour`.
2. **Tiered AI fallback** — never drop an incident, degrade the classifier:
   - Tier 1 (default): cloud provider (OpenAI/Anthropic).
   - Tier 2: auto-fallback to local Ollama at 80% of daily budget.
   - Tier 3: heuristic rules-only if Ollama is down or saturated.
3. **Priority queue**: critical severity + first-seen entities always use
   Tier 1 regardless of budget state, so a volumetric attacker cannot
   starve the analysis of a targeted attacker riding the noise.
4. **Reverse circuit breaker**: if hourly AI spend exceeds 5× the 7-day
   baseline, pre-emptively drop to Tier 2 until the spike clears.
5. **Metrics**: `innerwarden_ai_tokens_total{provider,tier}` +
   `innerwarden_ai_usd_spent_total` for dashboard drift detection.

Justification — worst-case math on current code (50 concurrent attackers):
- Cooldown prevents re-calls on same `action:detector:entity`; practical
  ceiling ~250 unique combinations/hour = ~$15/h on GPT-4o, ~$0 on Ollama.
- But: a backlog means real attackers can slip through undetected, which
  is the expensive failure mode. Fix the backlog, not the budget.

## Spec 027 candidate — Multi-agent AI + MITRE-tiered auto-response

Emerged from the same Caldera exercise conversation. Two linked gaps that
the current single-provider, per-tick-capped AI pipeline cannot close:

### Why one model is not enough

qwen2.5:1.5b (default local) cannot reason about proportional response:
kill process vs isolate container vs leave honeypot vs do nothing requires
understanding the kill-chain context, operator environment tolerance, and
blast radius. Asking it to decide an active-defense playbook is asking the
wrong cognitive tier.

### Proposed multi-agent architecture

| Role | Model tier | Job | Budget per call |
|------|------------|-----|-----------------|
| Triage | qwen 1.5b local | "Noise or signal?" — drops 95% before deeper analysis | ~3s / $0 |
| Analyst | qwen 3b local or GPT-4o-mini | "Which MITRE technique? Corroborating evidence?" | 5-30s / $0.001 |
| Responder | Cloud (Sonnet/Opus) | "Given chain + capability + policy, which playbook?" | 30-120s / $0.05 |

Responder only fires for **decisive actions** (block/kill/isolate), which
in current data is <0.5% of incidents. Daily budget stays under $1 even
under sustained attack — the cheap local Triage does the filtering. The
existing `playbook.rs` engine stays the execution layer; the Responder
LLM picks and parameterises a playbook instead of picking a single skill.

### Why auto-block on Caldera-confirmed techniques is tempting but dangerous

Caldera exercises real MITRE techniques, but a technique fire ≠ an attack.
Real-world FP risk varies wildly by TID:

| MITRE TID | FP risk | Safe default |
|-----------|---------|--------------|
| T1486 (encrypt files / ransomware) | low | auto-block + quarantine |
| T1496 + YARA xmrig match | low | auto-kill process |
| T1055 (injection into RWX anon mapping) | low | auto-kill |
| T1014 (kernel module load) | medium | require chain corroboration |
| T1098 (account modify) | medium | notify + corroboration |
| T1547 (boot/autostart) | high | notify only |
| T1033 (`whoami`) | extreme | observe, feed graph |
| T1059 (shell exec) | extreme | observe |

A single TID rarely justifies a block. A **correlation chain** does —
`whoami → find /etc/shadow → outbound C2 → insmod` is an attack regardless
of any single step being benign in isolation. The existing 47 CL-*
correlation rules already encode that pattern, but there is no tier-map
that says "for T1486, block on sight" vs "for T1059, observe forever".

### What's missing / to be specified

1. `rules/mitre_auto_policy.toml` — per-TID policy tier (block / kill /
   isolate / notify / observe) + `require_corroboration: bool` + minimum
   chain length. Operator-editable.
2. Multi-tier AI routing in `ai/mod.rs`: triage / analyst / responder
   roles with fallbacks. Promote to cloud only when triage says "real"
   and analyst says "needs action decision". Local-first; cloud is
   opt-in per-tier.
3. Operator UI on Threats tab: per-TID override ("In my env, T1014 is
   never benign because I never load kernel modules manually — auto-block").
4. Budget tie-in: the $-per-day cap from the "AI budget" follow-up above
   flows into Responder tier admission (Triage and Analyst stay uncapped
   because they are local / cheap).

### Why it is spec 027, not today

The existing correlation engine + playbook engine are the right primitives;
this spec just layers policy tiers and agent routing on top. No sensor
changes, no schema migrations. Expected scope: ~2 weeks of work, gated on
benchmarking the Triage/Analyst/Responder split against recorded real
incidents (we have the data from this exercise already — the observation
verification logs include the AI decisions we would use as ground truth).
